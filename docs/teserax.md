# Teserax

A graph-based thinking tool that transforms linear LLM chat into a visual, non-linear exploration space. Ask questions in a persistent chat lane, watch responses decompose into structured graph nodes on an interactive canvas, branch into parallel threads, merge insights, cross-link between reasoning paths, and summarize -- all with cloud persistence, multi-model LLM support, and bring-your-own-key for any provider.

## Architecture Overview

Teserax uses a **dual-lane response model**:

```
Lane A (Chat)                       Lane B (Graph)
┌─────────────────┐                ┌──────────────────────────────────┐
│  ChatPanel       │                │  GraphCanvas (React Flow)        │
│  - conversation  │  ──linked──▶  │  - structured nodes              │
│  - immutable raw │                │  - typed chunks (projection)     │
│  - message history│               │  - cross-links, swim lanes       │
└─────────────────┘                └──────────────────────────────────┘
```

Every LLM operation emits immutable chat messages (Lane A) in parallel with graph nodes (Lane B), linked via `projectedNodeIds`. The graph stores both `rawContent` (immutable provider response) and `projection` (derived `StructuredProjection`, re-extractable).

### Core Loop

```
User sends a message in the ChatPanel
  -> LLM responds (streamed via SSE, using server key or user's BYOK key)
  -> Response is projected into a StructuredProjection (Zod-validated typed blocks)
  -> Each block becomes a node on the graph canvas
  -> Structural edges (top/bottom handles) enforce strict DAG with cycle detection
  -> User can:
      - Query any node further (branching into a new swim lane thread)
      - Merge multiple nodes (converging)
      - Summarize a reasoning path (compressing)
      - Cross-link between branches (side handles -- non-structural annotations)
      - Accept, edit, or reject individual chunks before they join the graph
  -> The graph preserves full reasoning chains, execution metadata, and context lineage
  -> Context window management trims oldest messages at 80% token budget
  -> Auto-analyze toggle: LLM planner decomposes the query into a multi-step
     execution plan, then the pipeline engine runs all steps automatically
  -> Node chat modal: click any node to iterate in-place or spawn a child node
```

### Prompt Engineering

| Category | Details |
|----------|---------|
| **Prompt Registry** | Versioned prompt templates as data objects in `shared/prompts/`. 8 templates: `chat.default`, `extract.structure`, `op.decompose`, `op.merge`, `op.summary`, `op.crosslink`, `op.requery`, `orchestrator.plan`. Structured sections (task, context, constraints, instructions) with `renderTemplate()` interpolation and style modifiers. |
| **Canonical Schemas** | `shared/schemas/` with `StructuredProjection` (Lane B: typed blocks, topics, tags, artifacts, lineage). Per-operation schemas: `DecomposeOutput`, `MergeOutput` (agreements/conflicts/openQuestions), `SummaryOutput` (faithfulness), `CrosslinkOutput` (needsConfirmation). JSON Schema objects for OpenAI `response_format`. |
| **Validation + Corrective Retry** | `validateStructuredOutput()` with JSON parse, Zod validation, semantic checks. On failure, feeds schema errors back to LLM for a second attempt. Graceful fallback to `splitBullets()` heuristic with user notification. |
| **Context chaining** | Chat queries use thread message history; graph operations use `walkPathToRoot` for path-based context. `trimMessagesToFit` drops oldest messages at 80% token budget. Model-aware server guard. |
| **Crosslink context injection** | Cross-linked node content injected into LLM prompts without violating DAG structure. `gatherCrosslinkContext()` collects content from crosslinked nodes; `budgetCrosslinkContext()` allocates token budget. Crosslink context appears as a dedicated block in assembled prompts. |

### AI Orchestration

| Category | Details |
|----------|---------|
| **Auto-analyze toggle** | Prominent toggle in ChatPanel. When enabled, every chat message is routed through the orchestrator instead of creating a single query node. |
| **LLM planner** | `orchestrator.plan.v1` prompt asks the LLM to decompose any natural-language query into a 3–7 step structured JSON plan (`OrchestrationPlan`). Each step specifies a key, title, hue, dependencies, merge flag, and a detailed self-contained prompt. |
| **Pipeline engine** | `templatePipeline.ts` executes plans as a DAG of steps. Steps with no dependencies run in parallel; downstream steps receive upstream RESPONSE content via `buildPrompt()`. Supports both static registered pipelines and dynamically built plans from the orchestrator. |
| **Fallback plan** | If LLM planning fails (parse error, timeout, etc.), a hardcoded 3-step plan (key themes → detailed analysis → synthesis merge) ensures the user always gets a multi-node result. |
| **Thread auto-creation** | Each orchestration run creates a dedicated chat thread linked to the graph root, so orchestration messages don't pollute manual chat history. |
| **Progress tracking** | `OrchestrationStatus` with per-step state (`pending` / `running` / `done`), step count, and current step label. Rendered in ChatPanel as a live progress indicator. |

| Category | Details |
|----------|---------|
| **In-node iteration** | Click any node to open a full-height streaming chat modal (`NodeChatModal.tsx`). The modal uses the node's ancestry chain (`walkPathToRoot`) as conversation context. |
| **Completion actions** | Two buttons after each assistant response: **Apply Changes** (overwrites the node's content in-place) and **Generate New Node** (creates a child node via `createQuery`). |

## Quick Start

### Prerequisites

- Node.js 20+
- npm
- At least one LLM API key (OpenAI or Grok/xAI) -- OR users can bring their own via BYOK
- (Optional) Supabase project for cloud features

### Local Development

```bash
# Clone
git clone https://github.com/dhairyamishra/teserax.git
cd teserax

# Install dependencies
npm install

# Configure environment
cp .env.example .env
# Edit .env -- add your OPENAI_API_KEY and/or XAI_API_KEY

# Run both frontend and backend
npm run dev:full
# Frontend: http://localhost:8080  (proxies /api to backend)
# Backend:  http://localhost:3001
```

### With Supabase (cloud features)

1. Create a [Supabase](https://supabase.com) project
2. Apply the database migrations from `supabase/migrations/` in order (006 through 007) in the Supabase SQL Editor, plus set up core tables via the RPC functions
3. Add to `.env`:
   ```
   VITE_SUPABASE_URL=https://your-project.supabase.co
   VITE_SUPABASE_ANON_KEY=your-anon-key
   AUTH_DISABLED=false
   ```
4. For ES256 JWT verification, add:
   ```
   SUPABASE_JWT_PUBLIC_KEY={"kty":"EC","crv":"P-256",...}
   ```
   Get the JWK from Supabase Dashboard > Settings > API. Falls back to HS256 with `SUPABASE_JWT_SECRET` if not set.
5. (Optional) Enable Google OAuth in Supabase Dashboard > Authentication > Providers

### With Docker

```bash
npm run docker:dev     # Local development
npm run docker:prod    # Production with Cloudflare Origin SSL
```

## Roadmap & Issue Tracker

### Phase 2 -- Core UX Elevation (11 issues, all closed)

### Phase 3 -- LLM Engine Revolution (10 issues, all closed)

### Post-Phase: Monetization (implemented, then disabled)
Stripe payment integration with 3-tier subscription model (Free/Pro/Team), webhook handling, subscription middleware, admin email bypass, pricing page. Later disabled -- all features free while monetization is not permitted.

### v2.0 -- Architecture Overhaul (completed)
Major architectural changes post-Phase 5:

- **Dual-lane response model**: Chat panel (Lane A) + graph canvas (Lane B) with linked `projectedNodeIds`
- **Chat-first UI**: Three-panel layout (ChatPanel / GraphCanvas / Context Strip)
- **Prompt registry**: 8 versioned templates in `shared/prompts/` with `renderTemplate()` interpolation
- **Canonical schemas**: `StructuredProjection` and per-operation schemas in `shared/schemas/`
- **Corrective retry**: Validation pipeline feeds errors back to LLM for auto-correction
- **Context window management**: Token budget tracking with oldest-message trimming
- **Top-to-bottom tree layout**: Replaced left-to-right force simulation with deterministic tree
- **Swim lanes**: Multi-root support with visual lane dividers and drag clamping
- **Four-handle crosslinks**: DAG-safe structural edges (top/bottom) + annotation crosslinks (left/right)
- **Glassmorphism UI**: `backdrop-blur-md`, dynamic node sizing, compact operator pills, shimmer skeleton
- **Multimodal content**: Attachment support (image/audio/video/file) on nodes
- **TopBar model switcher**: Interactive provider-grouped dropdown
- **Default model**: Switched to grok-3-mini with new Grok models (4.1, 3, 3 Mini, 2)
- **Financial Analysis template**: Bull/bear case framework (now auto-executing pipeline)
- **Performance**: TTL caches, AbortController + timeouts, batched Zustand updates, streaming wired to execution slice
- **Production hosting**: EC2 + Cloudflare Origin Certificate (replaced Let's Encrypt/Certbot)
- **ES256 JWT**: Support for Supabase's newer ES256 auth tokens
- **Comprehensive testing**: 226 Vitest tests (was 38), mock infrastructure, 18 E2E spec files
- **Stripe disabled**: Billing gutted (returns 403), all features free
- **Sidebar redesign**: Slim ~220px read-only context strip with smart chat targeting
- **Rebrand**: DecompositionAI -> Teserax across entire codebase

### Post-v2.0 -- Orchestration & Node Interaction (completed)

- **Crosslink context injection**: Cross-linked node content injected into LLM prompts with token budgeting via `gatherCrosslinkContext()` and `budgetCrosslinkContext()`
- **Template pipeline engine**: Replaced 8 static templates with 3 auto-executing pipeline templates (Financial Analysis, Pros & Cons, Decision Matrix). DAG-based `templatePipeline.ts` with dependency tracking and parallel step execution
- **AI auto-orchestration**: LLM planner (`orchestrator.plan.v1`) decomposes any natural-language query into a 3–7 step structured execution plan. Dynamic pipeline builder converts plans into executable DAGs. Fallback 3-step plan on planner failure. Auto-analyze toggle in ChatPanel. Thread auto-creation and per-step progress tracking (`OrchestrationStatus`)
- **Node chat modal**: Click any node to open a streaming chat modal (`NodeChatModal.tsx`) for in-place iteration. Two completion actions: Apply Changes (update node) and Generate New Node (spawn child). Dedicated refinement system prompt
- **Hover-expand nodes**: CSS `group-hover` expansion replaces expand-on-select. QueryNode shows full text
- **LLM-generated metadata**: `nodeMetaGenerator.ts` background sidecar generates 3–6 word titles and one-sentence summaries for RESPONSE nodes, debounced at 400ms with AbortController cancellation
